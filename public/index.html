75XP Web API Reference

All request and response bodies are JSON unless noted otherwise. Endpoints that accept file uploads use multipart/form-data. Protected endpoints require a valid access token in the Authorization: Bearer <token> header. Tokens are issued on login and registration, and can be refreshed via /api/auth/refresh without re-authentication.

For the frontend team

Topic Detail
Access token Short-lived JWT. Returned in the response body. Include as Authorization: Bearer <token> on all protected requests.
Refresh token Longer-lived token. Set and rotated as an HttpOnly cookie refreshToken. Not accessible via JavaScript. Rotated on every successful refresh.
Signed URLs avatar_url and progress_pic_url in responses are pre-signed URLs valid for 7 days. Do not store them permanently — re-fetch when needed.
File uploads POST /api/auth/register, PATCH /api/profile, and POST /api/days accept multipart/form-data, not JSON. Send all fields as form fields and the file under its designated key.
Session invalidation Changing a password (PATCH /api/profile/password) and logging out (POST /api/auth/logout) both invalidate all existing sessions. The user must log in again.
Token expiry When the access token expires, call POST /api/auth/refresh using the cookie. If the refresh token is also expired, the user must log in again from scratch.
Login error Login returns the same 401 and the same error message for both a wrong password and an unknown email. This is intentional.
Weekly summary On every 7th day, POST /api/days generates an AI summary of the preceding 7 days and returns it in weekly_summary. All other days return "weekly_summary": null.

Health check

Returns the current API version. Also use this to verify the API is reachable. No authentication required.

Response 200 OK

}

Authentication

The access token is returned in the response body and must be included in the Authorization header for all protected endpoints.

POST /api/auth/register

Creates a new user account. Accepts multipart/form-data. On success, returns the new user's details along with an access token and sets a refresh token cookie.

Request Body — multipart/form-data
Field Type Required Notes
email string Yes Must be a valid email address.
name string Yes Minimum 3 characters.
password string Yes See password rules.
avatar_key string Yes One of: avatar_1.png, avatar_2.png, avatar_3.png, avatar_4.png
Response 201 Created 
{
  "user": {
    "id": "<uuid>",
    "name": "Test User",
    "email": "testuser@example.com",
    "avatar_url": "<signed-url>"
  },
  "accessToken": "<access-token>"
}

avatar_url is a pre-signed URL valid for 7 days. A refreshToken cookie is also set.

Error Responses
Status Condition Error message
400 A required field is missing or empty <field> cannot be empty
400 Email is not a valid format Invalid email address
400 Name is shorter than 3 characters Name must be at least 3 characters
400 Password does not meet complexity requirements Password must be at least 8 characters and contain at least: one uppercase letter, one lowercase letter, one digit and one special character
400 avatar_key is not one of the accepted values Avatar key not recognised

POST /api/auth/login

Authenticates an existing user with their email and password. Returns the user's details, a fresh access token, and sets a refresh token cookie.

Request Body — application/json
Field Type Required
email string Yes
password string Yes
Response 200 OK 
{
  "user": {
    "id": "<uuid>",
    "name": "Test User",
    "email": "testuser@example.com",
    "avatar_url": "<signed-url>"
  },
  "accessToken": "<access-token>"
}

avatar_url is a pre-signed URL valid for 7 days. A refreshToken cookie is also set.

Error Responses
Status Condition Error message
401 Email not found or password is incorrect Invalid email or password

Both cases return the same message intentionally to prevent account enumeration.

POST /api/auth/refresh

Issues a new access token using a valid refresh token. The refresh token is read from the request cookie and replaced with a new one on success.

Request Cookie 
refreshToken=<valid-refresh-token>
Response 200 OK 
{
  "accessToken": "<new-access-token>"
}

A new refreshToken cookie is set, replacing the old one.

Error Responses
Status Condition Error message
401 No refresh token cookie present No refresh token provided
401 Refresh token is expired or has been tampered with Refresh token expired or invalid. Please log in again.

POST /api/auth/logout

Logs the user out by revoking all active refresh tokens for their account. The client should discard the access token after calling this endpoint. No request body is required.

Request Headers 
Authorization: Bearer <valid-access-token>
Response 200 OK 
{
  "message": "Logged out successfully."
}

User Profile

All profile endpoints require a valid access token in the Authorization header using the Bearer scheme.

Authorization Header Format 
Authorization: Bearer <access-token>
Common Authorization Errors (apply to all protected endpoints)
Status Condition Error message
401 No Authorization header sent No authorization header provided
401 Header present but token is missing the Bearer scheme Unrecognized authorization header
401 Token is structurally malformed Invalid access token
401 Token is valid but has expired Access token expired — use /api/auth/refresh to get a new one

GET /api/profile

Returns the full profile of the currently authenticated user.

Response 200 OK 
{
  "user": {
    "id": "<uuid>",
    "name": "Test User",
    "email": "testuser@example.com",
    "avatar_url": "<signed-url>"
  }
}

avatar_url is a pre-signed URL valid for 7 days.

PATCH /api/profile

Updates the authenticated user's display name and/or avatar. Accepts multipart/form-data. Only the fields you include will be updated. Returns the full updated profile.

Request Body — multipart/form-data
Field Type Required Notes
name string No Minimum 3 characters.
avatar_pic file No Replaces the current avatar. Uploaded as a file field.
Response 200 OK 
{
  "user": {
    "id": "97e1bffd-3dc5-4eda-83c9-606782836b17",
    "name": "Test User 1",
    "email": "testuser@example.com",
    "avatar_url": "<signed-url>"
  }
}

avatar_url is a pre-signed URL valid for 7 days.

PATCH /api/profile/password

Changes the authenticated user's password. The current password must be provided for verification. All existing sessions are invalidated on success — the user must log in again.

Request Body — application/json
Field Type Required
current_password string Yes
new_password string Yes — must meet password rules
Response 200 OK 
{
  "message": "Password changed successfully. Please log in again."
}
Error Responses
Status Condition Error message
400 Provided current password does not match Current password is incorrect
400 New password does not meet complexity requirements Password must be at least 8 characters and contain at least: one uppercase letter, one lowercase letter, one digit and one special character

Daily Check-ins

All daily check-in endpoints are protected. Each user has at most one record per day number (1–75), enforced at the database level. Days must be created in order — a day cannot be created unless the previous day already exists.

Endpoints that accept a file upload use multipart/form-data. The progress_pic_url in responses is a pre-signed URL valid for 7 days.

GET /api/days

Returns all daily check-in records for the authenticated user alongside any weekly summaries generated so far. Days are ordered by day_number ascending.

Request Headers 
Authorization: Bearer <valid-access-token>
Response 200 OK 
{
  "days": [
    {
      "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "day_number": 1,
      "diet_adhered": true,
      "outdoor_workout_completed": true,
      "indoor_workout_completed": true,
      "water_consumed": true,
      "pages_read": true,
      "mood_rating": 4,
      "achievements": "Completed first day strong. Stuck to meal prep and both workouts.",
      "challenges": "Morning workout felt rough. Body not used to early wake-ups.",
      "next_day_focus": "Set alarm 15 minutes earlier to give more transition time.",
      "progress_pic_url": "<signed-url>"
    }
  ],
  "weekly_summaries": [
    {
      "week": 1,
      "summary": "<weekly summary text>"
    },
    {
      "week": 2,
      "summary": "<weekly summary text>"
    }
  ]
}

Both arrays are empty if no days have been logged yet. weekly_summaries only contains entries for completed weeks.

GET /api/days/:day_number

Returns the check-in record for a specific day. day_number is passed as a URL parameter and must be between 1 and 75 inclusive. The response also includes the weekly summary for that week if one has been generated; otherwise weekly_summary is null.

Request Headers 
Authorization: Bearer <valid-access-token>
Response 200 OK 
{
  "day": {
    "day_number": 12,
    "diet_adhered": true,
    "outdoor_workout_completed": true,
    "indoor_workout_completed": true,
    "water_consumed": true,
    "pages_read": true,
    "mood_rating": 4,
    "achievements": "Six weeks complete. Mental toughness is the biggest gain so far.",
    "challenges": "Social pressure from family questioning the diet again.",
    "next_day_focus": "Hold the boundary without over-explaining.",
    "progress_pic_url": "<signed-url>"
  },
  "weekly_summary": null
}
Error Responses
Status Condition Error message
400 day_number is outside the 1–75 range Day number must be between 1 and 75 inclusive
404 Day is within range but has not been logged yet day number :day_number does not exist

POST /api/days

Creates a new daily check-in. Accepts multipart/form-data to support the optional progress photo upload. Days must be logged in order — day 2 cannot be submitted before day 1 exists. On every 7th day, an AI-generated weekly summary is produced and returned alongside the day.

Request Headers 
Authorization: Bearer <valid-access-token>
Request Body — multipart/form-data
Field Type Required Notes
day_number integer Yes 1–75 inclusive
diet_adhered boolean No Must be a boolean, not a string
outdoor_workout_completed boolean No Must be a boolean, not a string
indoor_workout_completed boolean No Must be a boolean, not a string
water_consumed boolean No Must be a boolean, not a string
pages_read boolean No Must be a boolean, not a string
mood_rating integer No 1–5 inclusive
achievements string No 10–500 characters
challenges string No 10–500 characters
next_day_focus string No 10–500 characters
progress_pic file No Image file. Uploaded as a file field.
Response (days 1–6, 8–13, etc.) 201 Created 
{
  "day": {
    "day_number": 1,
    "diet_adhered": true,
    "outdoor_workout_completed": true,
    "indoor_workout_completed": true,
    "water_consumed": true,
    "pages_read": true,
    "mood_rating": 4,
    "achievements": "Completed first day strong. Stuck to meal prep and both workouts.",
    "challenges": "Morning workout felt rough. Body not used to early wake-ups.",
    "next_day_focus": "Set alarm 15 minutes earlier to give more transition time.",
    "progress_pic_url": "<signed-url>"
  },
  "weekly_summary": null
}
Response (every 7th day — day 7, 14, 21…) 201 Created 
{
  "day": {
    "day_number": 7,
    "diet_adhered": true,
    "outdoor_workout_completed": true,
    "indoor_workout_completed": true,
    "water_consumed": true,
    "pages_read": true,
    "mood_rating": 4,
    "achievements": "Completed first day strong. Stuck to meal prep and both workouts.",
    "challenges": "Morning workout felt rough. Body not used to early wake-ups.",
    "next_day_focus": "Set alarm 15 minutes earlier to give more transition time.",
    "progress_pic_url": "<signed-url>"
  },
  "weekly_summary": {
    "week": 1,
    "summary": "<AI-generated summary of the past 7 days>"
  }
}

The weekly summary is generated at the time of the day 7, 14, 21 … submission. It covers the full preceding week and includes analysis of consistency, highlights, and suggested focus areas for the week ahead.

Error Responses
Status Condition Error message
400 day_number is missing day_number cannot be empty
400 day_number is outside the 1–75 range Day number must be between 1 and 75 inclusive
400 Previous day does not exist (sequential order enforced) Day N must be completed before creating day N+1
400 A boolean field is sent as a string (e.g. "true" instead of true) <field> must be a boolean
400 mood_rating is outside the 1–5 range Mood rating must be between 1 and 5 inclusive
400 achievements, challenges, or next_day_focus is under 10 or over 500 characters <field> must be at least 10 characters and must not exceed 500 characters

DELETE /api/days/:day_number

Deletes the check-in record for a specific day. day_number must be between 1 and 75. Returns no body on success.

Request Headers 
Authorization: Bearer <valid-access-token>
Response 204 No Content Error Responses
Status Condition Error message
400 day_number is outside the 1–75 range Day number must be between 1 and 75 inclusive
404 Day is within range but has not been logged day number :day_number does not exist

Milestones

Milestones are awarded automatically when a user completes day 25, 50, or 75 with all tasks checked off. They cannot be created or deleted manually.

GET /api/milestones

Returns all milestones earned by the authenticated user. Returns an empty array if none have been awarded yet.

Request Headers 
Authorization: Bearer <valid-access-token>
Response 200 OK 
[
  {
    "badge_type": "bronze",
    "awarded_at": "2024-09-25T21:00:00.000Z"
  },
  {
    "badge_type": "silver",

Input Validation Rules

Password Requirements

All passwords (on registration and password change) must satisfy the following:

  • At least 8 characters long
  • At least one uppercase letter (A–Z)
  • At least one lowercase letter (a–z)
  • At least one digit (0–9)
  • At least one special character (e.g. !, $, @, #)

General Field Rules

Field Rule
email Must be a valid email address format (e.g. user@example.com)
name Minimum 3 characters
avatar_key Must be one of: avatar_1.png, avatar_2.png, avatar_3.png, avatar_4.png
mood_rating Integer between 1 and 5 inclusive
achievements, challenges, next_day_focus Between 10 and 500 characters
Boolean fields (task completion) Must be a JSON boolean (true / false), not a string
